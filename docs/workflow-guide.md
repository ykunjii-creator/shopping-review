# 워크플로우 쉽게 이해하기

> 이 문서는 프로젝트를 **처음 보는 사람**을 위한 안내입니다. 코드를 몰라도, 각 단계가 "무엇을, 왜" 하는지 그림처럼 따라올 수 있게 썼습니다.

---

## 한 줄 요약

> **올리브영 상품 리뷰를 자동으로 모아 → AI가 "이건 제품 결함이다/단순 불만이다" 분류하고 → 믿을 만한지 두 번 더 검증한 뒤 → 엑셀에 정리하고 → 심각한 결함이면 담당자에게 메일로 알려주는 시스템.**

사람이 손으로 하면 몇 시간 걸릴 "리뷰 읽고 분류하고 보고하는" 일을, **로봇(수집·기록·메일)** 과 **AI(판단)** 가 나눠서 자동으로 합니다.

---

## 등장인물 (역할 분담)

| 누구 | 정체 | 하는 일 | 비유 |
|:--|:--|:--|:--|
| 🤖 **RPA (UiPath)** | 윈도우에서 도는 로봇 | 브라우저 열어 리뷰 긁기, 엑셀 쓰기, 메일 보내기, 결과 점검 | **손과 발** |
| 🧠 **AI 서버 (Python)** | 컴퓨터 안에서 도는 분석 프로그램 | 리뷰 글을 읽고 "무슨 문제인지" 판단 | **두뇌** |

둘은 **인터넷 주소(`localhost:8000`)로 메시지를 주고받으며** 협업합니다. (서로 직접 붙어있지 않아서, 한쪽을 고쳐도 다른 쪽은 그대로 둘 수 있음.)

---

## 전체 흐름 한눈에

```
[올리브영 페이지]
      │  ① 리뷰 수집 (로봇이 브라우저로)
      ▼
   리뷰 목록 (글·별점·날짜)
      │  ② AI 서버로 전송
      ▼
   🧠 AI 분석 + 3중 검증  ─────────────┐
      │  ③ 분류 결과 + 종합 요약        │  (이 단계가 핵심!)
      ▼                                 │
   ④ 엑셀에 한 줄씩 기록 (Master_Log)   │
      │                                 │
      ├─ 제품결함 있으면 → ⑤ 긴급 메일   │
      ├─ ⑥ 종합 리포트 엑셀 생성         │
      └─ ⑦ 잘 됐는지 자동 점검 (검증)    │
                                        ┘
```

## 단계별 설명

### ① 리뷰 수집 — `Scrape_Reviews`
**무엇을 하나:** 로봇이 **크롬 브라우저를 직접 열어** 올리브영 상품 페이지로 들어가, 리뷰를 긁어옵니다.

- **왜 어려웠나:** 올리브영 리뷰는 화면에 보여도 **특수한 구조(Shadow DOM)** 안에 숨어 있고, **스크롤해야 다음 리뷰가 로딩**되는 구조(무한스크롤)입니다. 일반적인 "표 긁기" 기능으론 1건밖에 못 잡습니다.
- **어떻게 풀었나:** 페이지 안에 작은 **자바스크립트(JS)** 를 집어넣어, 숨은 구조를 뚫고 리뷰를 읽습니다. 그리고 **조금씩 스크롤 → 새로 뜬 리뷰 수집 → 중복 제거**를 반복해, 더 안 늘어날 때까지 모읍니다.
- **결과물:** 리뷰 목록 (각 줄에 `리뷰ID · 본문 · 별점(1~5) · 작성일`).
- 리뷰가 **0건이면** 여기서 안전하게 종료합니다(괜히 다음 단계로 안 넘어감).

> 💡 다른 상품을 보고 싶으면 `in_GoodsUrl` 값만 바꾸면 됩니다. (README의 "대상 URL 바꾸기" 참고)

### ② AI 서버로 전송 — `HTTP POST /analyze-reviews`
**무엇을 하나:** 모은 리뷰 목록을 **AI 서버에 통째로 보냅니다** (인터넷 주소로 택배 보내듯).

- 보내는 형식: `{ 배치번호, 리뷰들[] }`
- AI 서버는 리뷰를 **한 건씩 차례로** 분석합니다. 그래서 리뷰가 많으면 시간이 좀 걸립니다.

### ③ AI 분석 + 3중 검증 — *AI 서버 안에서 (가장 중요)*
**무엇을 하나:** 이 프로젝트의 **핵심**. AI 결과를 그냥 믿지 않고, **3번 거릅니다.**

리뷰 한 건이 거치는 3개의 관문:

1. **🧠 1관문 — AI가 분석** : 리뷰 글을 읽고 판단합니다.
   - 카테고리(제품결함/배송문제/단순불만/긍정), 30자 요약, 담당 부서·메일, 긴급도, **확신도(0~1)**, 판단 근거.
   - 필요하면 스스로 웹 검색도 하고, 부서 매핑표도 찾아봅니다.
2. **📏 2관문 — 규칙 검사 (Rule-based)** : 사람이 만든 **9가지 상식 규칙**으로 점검.
   - 예) "요약이 30자를 넘으면 안 됨", "긍정인데 긴급도 높음은 모순", "결함인데 별점 정보와 안 맞음" 등.
   - 하나라도 어기면 → **'미분류'로 강등**하고 3관문은 건너뜀.
3. **⚖️ 3관문 — 다른 AI가 채점 (LLM-as-a-Judge)** : **1관문과 다른 AI**가 그 판단을 1~10점으로 채점.
   - **7점 미만이면 → '미분류'로 강등.**

> 이렇게 **AI(분석) → 규칙(검사) → AI(재채점)** 를 통과한 것만 "verified(검증됨)" 도장을 받습니다. 믿을 수 없는 판단은 자동으로 '미분류'로 빠져, 잘못된 경보를 막습니다.

- 마지막으로 전체를 **종합 분석**합니다: 카테고리 분포, 결함 유형 묶음, 자주 나온 키워드, 통과율, 핵심 발견 등.
- **결과물:** 리뷰별 분석 결과 목록 + 종합 요약 + **결함 개수(`defect_count`)**.

### ④ 엑셀에 기록 — `Process_Results`
**무엇을 하나:** AI 결과를 받아 **`Master_Log.xlsx`** 에 **한 줄씩 누적** 저장합니다 (12개 칸: 분석일시·리뷰ID·원본텍스트·별점·작성일·카테고리·요약·담당부서·담당메일·긴급도·확신도·판단근거).

- **똑똑한 점:** 기존 파일이 **헤더(제목 줄) 없이 망가져 있어도**, 자동으로 알아채고 제대로 다시 만듭니다(self-heal). → 다음 단계 검증이 실패하지 않게.

### ⑤ 긴급 메일 — `Send_Urgent_Mail` *(조건부)*
**무엇을 하나:** **제품결함이 1건이라도 있으면**, 담당자에게 **Gmail로 긴급 메일**을 보냅니다 (결함만 모은 엑셀 첨부).

- 제목 예: `[긴급] 제품결함 N건 탐지 - batch ...`
- **결함이 0건이면 이 단계는 그냥 건너뜁니다** (메일 안 옴이 정상).
- 보내려면 Gmail **앱 비밀번호**가 환경변수에 설정돼 있어야 합니다. 없으면 경고만 남기고 건너뜀(전체 흐름은 계속).

> 💡 메일이 안 온다면? 대개 **결함이 0건**이거나, **앱 비밀번호 미설정** 때문입니다. (README "Gmail SMTP 설정" 참고)

### ⑥ 종합 리포트 — `Generate_Summary_Report`
**무엇을 하나:** ③의 종합 분석을 보기 좋은 **엑셀 리포트(4개 시트)** 로 만듭니다. → `Data/Reports/종합_분석_리포트_*.xlsx`

- 시트: **검증 요약 · 결함 유형 · 카테고리 분포 · 핵심 발견**.

### ⑦ 실행 검증 — `Verify_RPA_Results`
**무엇을 하나:** "여기까지 제대로 됐나?"를 **로봇이 스스로 점검**합니다.

- 점검 항목: `Master_Log.xlsx` 존재 여부, **12개 칸 다 있는지**, 빈 칸(카테고리/요약) 없는지, 누적 행 수, 메일 발송 여부 등.
- 결과를 **`Logs/execution_log.txt`** 에 `[VERIFY] PASS` 또는 `FAIL` 로 남깁니다.
- 심각한 문제는 `ERROR`(→ FAIL), 사소한 건 `WARNING`(→ PASS(경고))으로 구분.

## 용어 미니 사전

| 용어 | 쉬운 설명 |
|:--|:--|
| **카테고리** | 리뷰를 4종으로 분류: `제품결함`(불량·하자) · `배송문제`(늦음·파손) · `단순불만`(취향·기대차이) · `긍정`(만족). 못 믿겠으면 `미분류`. |
| **확신도(confidence)** | AI가 자기 판단을 얼마나 믿는지 (0~1). 낮으면 사람이 한 번 더 봐야 함. |
| **긴급도(urgency)** | 얼마나 급한 사안인지 (high/medium/low). |
| **검증됨(verified)** | 3중 관문을 다 통과한 신뢰 가능한 결과. |
| **미분류** | 규칙 위반이나 낮은 점수로 강등된, 신뢰 보류 상태. |
| **Shadow DOM** | 웹페이지 안에 "숨겨진 방"처럼 들어있는 구조. 일반 방법으로 못 들여다봐서 JS로 뚫음. |
| **무한스크롤** | 끝까지 스크롤할 때마다 다음 내용이 계속 로딩되는 방식. |
